How did viewing a diff between two versions of a file help you see the bug that
was introduced?

    Строка с багом отличалась от других строк, которые были найдены программой.
	Одно из внесенных в код изменений отличалось от других изменений. Программа
	лишь нашла все внесенные в код изменения. Кроме этого потребовалось сравнить
	изменения между собой, чтобы найти баг.

How could having easy access to the entire history of a file make you a more
efficient programmer in the long term?

    Любые изменения в коде могут быть отменены. Появляется свобода дальнейшего
	развития кода. В любой момент времени есть возможность вернуться к предыдущей
	версии. Или наоборот от предыдущей версии к более новой. Можно вносить
	любые изменения в код без страха что-либо сломать, т.к. всегда можно
	сделать шаг назад к версии, когда код работал. Вооруженный полной 
	историей кода проекта программист как бы выходит из плоскости кода.
	Он начинает смотреть на код в дополнительном измерении - во времени.
	Т.е. конкретная (текущая) версия кода воспринимется как временной срез 
	проекта, как конкретное воплощение в виде кода заложенных в проект идей
	и философии. При этом другие воплощения продолжают существовать, давая
	возможность в любой момент вермени вернуться к ним.
	

What do you think are the pros and cons of manually choosing when to create a
commit, like you do in Git, vs having versions automatically saved, like Google
docs does?

	Только сам программист может выявить логически верную точку коммита.
	Это и плюс и минус. Так как он может и упустить удачный момент коммита.
	Автоматизация коммита лишает его логического смысла, но дает возможность
	сохранить весь процесс создания кода. Почти что "видео"
    Неполохо бы иметь обе возможности. С одной стороны оставлять за собой
	выбор точки коммита для логиченого деления процесса создания кода на этапы.
	С другой стороны, почему бы не дать системе возможность периодически делать
	снимки текущего сосотояния проекта. Так, на всякий пожарный случай.

Why do you think some version control systems, like Git, allow saving multiple
files in one commit, while others, like Google Docs, treat each file separately?

    Одни программы контроля версий, например Git,
	ориентированны для работы с программными проектами,
	где часто возникает необходимость изменять два и более файлов параллельно
	для того, чтобы иметь, например, возможность успешно выполнить
	компиляцию кода в исполняемый файл.
	Другие, например Google Docs, ориентированы для работы с произвольным
	массивом текстовых документов, не имеющих такой тесной связи, как
	файлы программного проекта. Здесь более простое автоматизированное
	сохранение снимка каждого измененного файла вполне достаточно.
	

How can you use the commands git log and git diff to view the history of files?

    git log позволяет увидеть весь процесс разработки проекта во времени разбитый на коммиты.
	В логе для каждого коммита указан его id, автор, дата и время, когда коммит был принят, а
	также описание (сообщение) коммита.
	Опция --stat кроме этого указывает на количество и тип изменений в файле.
	git diff позволяет сравнить файлы из коммитов и увидеть все отличия между всеми файлами коммитов.

How might using version control make you more confident to make changes that
could break something?

    Прежде, чем менять что-либо в коде необходимо сделать коммит. Это позволит вернуться
	к этому состоянию кода если это потребуется в будущем.
